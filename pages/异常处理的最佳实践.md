- 建立统一分类的异常字典
- 同类异常的定义是可以扩展的
- 定义异常的严重程度
	- Fatal -> 重大问题
	- Error -> 资源或需求得不到满足
	- Warning -> 不一定是个问题但还是需要引起注意
	- Info -> 只是一个输出信息
	- Debug -> 调试使用。
- 输出的异常日志使用错误码，而不是错误信息。输出使用统一的格式。错误码不一定是数字，而是可以便于人读懂的短语。即便于人理解，也便于自动化程序对日志的分析
- 忽略的异常最好有日志输出
- 同一处不停报错的异常，限制输出的次数。避免淹没其他日志和导致日志文件过大
- 不要使用异常来控制（处理）业务逻辑
- 对于同类的异常，使用同样的模式来处理
- 无论在哪处理，尽可能的返回原始异常
- 处理异常时，要清理已分配的资源（RAII，Resource Acquisition Is Initialization）
- 尽量不要在循环体中处理异常
- 不要把大量代码放入try语句中，try语句应该是完成一个简单的任务
- 为定义的异常提供一个清楚明白的文档，并附有相应的示例
- 对于异步，推荐使用Promise模式处理异常
- 对于分布式，推荐使用APM（Application Performance Management）软件，使用Zipkin类似的服务调用跟踪和分析
